mod_audio_fork

Full-duplex audio bridge between FreeSWITCH and an AI voice bot over WebSocket.

FreeSWITCH 1.10 libwebsockets SpeexDSP 8 kHz → 16 kHz PCM16 / Binary WS

Background

mod_audio_fork is a custom FreeSWITCH module that bridges telephony audio to an AI voice pipeline running as a FastAPI/Pipecat service. Unlike media bugs that only tap audio passively, this module creates a full-duplex path: caller audio is streamed upstream to the bot for speech-to-text, and TTS audio synthesized by the bot flows back downstream and is injected into the RTP stream heard by the caller.

The module registers a dialplan application (audio_fork) which, when executed, attaches a media bug to the call and opens a WebSocket connection to the bot's URL. All audio transport and WebSocket lifecycle management is handled by the companion C++ library audio_pipe (built on libwebsockets), while mod_audio_fork.c handles FreeSWITCH integration.

Design Philosophy

The dominant pattern for integrating AI into telephony platforms is the webhook architecture: the media engine captures audio, serializes it into an HTTP request, waits for the AI service to respond, and plays back the result. Every decision the AI makes involves a full HTTP round-trip, and the media engine itself has no awareness of conversational state.

"When inference lives in the application layer, every decision round-trips through HTTP and behavioral enforcement gets handed to a stateless text predictor. That is Prompt and Pray." — Brian West, co-creator of FreeSWITCH, Head of Developer Experience @ SignalWire

mod_audio_fork takes a different approach. Rather than sitting above the media engine as an orchestration layer connected via HTTP, it compiles directly into FreeSWITCH as a native .so module and attaches to the RTP stack through a media bug running on the RTP thread itself.

Webhook architecture vs. kernel-level transport

Webhook / HTTP — "Prompt and Pray"

Audio captured by the media engine is forwarded over HTTP. Every AI decision requires a round-trip. The media engine is stateless between requests.

RTP → FreeSWITCH
  → HTTP POST (audio chunk)
    → AI service
  ← HTTP response (TTS audio)
→ RTP playback
mod_audio_fork — kernel-level transport

The media bug runs on the RTP thread. Audio flows through a persistent binary WebSocket with no HTTP framing, no per-frame serialization, no stateless round-trips.

RTP thread
  → media bug (C, zero-copy)
    → binary WS frame (PCM16)
      → AI pipeline (persistent)
    ← binary WS frame (TTS PCM16)
  ← WRITE_REPLACE (direct RTP inject)

What this means in practice

The transport layer imposes no HTTP overhead, no request serialization, and no per-utterance connection setup. The media bug's READ_REPLACE and WRITE_REPLACE callbacks are invoked directly by the FreeSWITCH core on every 20 ms RTP tick. Audio bytes move from the RTP stack into the WebSocket send buffer in the same thread, without a copy through an HTTP body.

State — reconnection attempts, ring buffer depth, jitter window, and optional RX CNG bridging — is maintained in C structs inside the running process, not reconstructed from a request payload on each call. The module enforces behavior in code: a disconnected bot triggers typed state transitions (AP_EVENT_RECONNECTING → AP_EVENT_DISCONNECTED) that drive typed C actions (hangup, silence fill), not prompt re-evaluation.

On the role of external inference STT (Deepgram), LLM, and TTS (Cartesia) necessarily live outside the FreeSWITCH process — they are cloud services that cannot be embedded in the media engine. mod_audio_fork represents the correct architecture given that constraint: the transport and behavioral enforcement layer is at the kernel level; only the inference boundary itself crosses a process line, and it does so over a persistent low-latency binary channel, not over per-request HTTP.

Features

  • Full-duplex audio: simultaneous RX (caller → bot) and TX (bot → caller) paths.
  • Automatic upsampling from 8 kHz (SIP/RTP) to 16 kHz (bot/STT) via SpeexDSP resampler (quality 4).
  • Session metadata frame sent as the first WebSocket TEXT message before any audio, allowing the bot to identify the session without parsing the URL.
  • Custom HTTP headers on the WebSocket handshake (e.g. Authorization: Bearer …, tenant IDs).
  • Configurable automatic reconnection with exponential backoff (100 ms → 5 000 ms).
  • Silence fill on the TX path when the bot is not sending audio (prevents underrun in RTP write frames).
  • Adaptive jitter buffer on the RX path.
  • Optional RX CNG bridging controlled by AF_RX_CNG_BRIDGE_ENABLE (default: disabled).
  • Production metrics: underruns, ring drops, send-queue drops, reconnection count, RX fragment bytes.
  • Graceful teardown: ap_cancel_service() wakes the LWS thread immediately via the internal pipe/eventfd, avoiding 15-second teardown delays.
  • Thread-safe design — media bug callbacks run on the RTP thread; the LWS service runs on a dedicated per-session thread.

Thread Model

Each active call managed by mod_audio_fork uses three execution contexts:

Thread 1
Dialplan app
audio_fork application. Runs switch_core_session_read_frame() loop as the media clock. Blocks until the call ends, then triggers teardown.
Thread 2
LWS service
Per-session thread calling ap_service() every 5 ms. Drives WebSocket I/O, reconnection timers, and the WS CLOSE flush on teardown.
Thread 3 (FS core)
RTP / media bug
READ_REPLACE pushes caller PCM to the bot. WRITE_REPLACE pulls bot TTS PCM from the ring buffer and writes it to the RTP stream.
Invariant Media bug callbacks (ap_on_rx_frame / ap_on_tx_frame) are called exclusively from the FreeSWITCH RTP thread. They must never block.

The WRITE_REPLACE callback always writes exactly FS_FRAME_SAMPLES = 160 samples (20 ms at 8 kHz). When the bot has not sent audio, zero/silence fills the frame so the RTP stream stays continuous.

WebSocket Protocol

mod_audio_fork uses two WebSocket frame types, following the same convention as other media-over-WebSocket protocols: TEXT frames carry structured control/metadata messages; BINARY frames carry raw PCM16 audio.

Frame typeDirectionContentWhen
TEXTFS → BotJSON session metadataFirst frame, immediately after ESTABLISHED
BINARYFS → BotPCM16 LE, 16 kHz, monoEvery RTP tick (~20 ms) while connected
BINARYBot → FSPCM16 LE, 8 kHz, monoTTS audio chunks from the bot
Frame ordering The metadata TEXT frame is always the first message on the WebSocket. Audio BINARY frames begin immediately after. Bot implementations must not send any BINARY audio before the TEXT frame has been sent by FreeSWITCH.

Session Metadata

On connection establishment, mod_audio_fork sends a JSON TEXT frame that allows the bot to identify the session and configure its pipeline without any URL parsing. 

{
  "uuid":            "ca7b176f-d394-49bd-a168-071b27e11a68",
  "direction":       "inbound",
  "caller":          "1000",
  "destination":     "510",
  "sip_sample_rate": 8000,
  "ws_sample_rate":  16000,
  "plan":            "premium",
  "language":        "pt-BR"
}
FieldTypeDescription
uuidstringFreeSWITCH channel UUID. Uniquely identifies this call leg.
directionstring"inbound" or "outbound" from the SIP perspective.
callerstringCaller-ID number (ANI).
destinationstringDialed number (DNIS).
sip_sample_rateintegerSample rate of the native RTP stream (always 8000 for G.711/G.729).
ws_sample_rateintegerSample rate of the audio sent over WebSocket (16000 after upsampling).
planstringCustom field — tenant plan, passed from the dialplan variable.
languagestringCustom field — BCP-47 language tag used to configure STT/TTS on the bot.

Additional custom fields can be injected by modifying the metadata JSON construction in mod_audio_fork.c before calling ap_create(). The pointer passed as metadata_json in AudioPipeConfig must remain valid until the AP_EVENT_CONNECTED event fires.

Audio Frames

All audio is transmitted as raw signed 16-bit little-endian PCM samples with no headers, packetization markers, or RTP encapsulation.

PropertyRX (FS → Bot)TX (Bot → FS)
EncodingPCM16 LEPCM16 LE
Sample rate16 000 Hz8 000 Hz
ChannelsMonoMono
Frame size~320 samples / 20 msVariable (ring buffer absorbs jitter)
WS frame typeBINARYBINARY
Bot implementation note The bot does not need to time its TX frames — the ring buffer and adaptive jitter buffer in audio_pipe handle reordering and rate matching. Send TTS audio chunks as they become available from the TTS service.

Connection Events

audio_pipe fires typed events on state transitions. These are handled by ap_event_handler() inside mod_audio_fork.c.

AP_EVENT_CONNECTED

WebSocket handshake completed successfully. Metadata is scheduled as the first WS TEXT frame and audio begins immediately after. Audio frames may flow in both directions.

AP_EVENT_RECONNECTING

The WebSocket was lost and a reconnection attempt is in progress. This event fires on every attempt when max_reconnect_attempts > 0. The call is not hung up. Audio will resume automatically when AP_EVENT_CONNECTED fires again.

AP_EVENT_DISCONNECTED

Final disconnection: either no reconnection was configured, or all reconnection attempts were exhausted. mod_audio_fork responds by calling switch_channel_hangup() with cause DESTINATION_OUT_OF_ORDER.

Dialplan Application

The module registers a single dialplan application named audio_fork. It is invoked with a data string containing the WebSocket URL and optional parameters.

Syntax

<action application="audio_fork" data="<ws-url> [max_reconnects=N]"/>

Parameters

ParameterDefaultDescription
ws-url—WebSocket URL of the bot endpoint. Supports ws:// and wss://. The UUID or other routing tokens can be embedded in the path.
max_reconnects00 = no reconnection. N = up to N attempts. -1 = infinite reconnection.

Examples

<!-- Basic: no reconnection -->
<action application="audio_fork"
        data="ws://127.0.0.1:8000/ws/${uuid}"/>

<!-- With metadata fields in URL path -->
<action application="audio_fork"
        data="ws://127.0.0.1:8000/ws/${uuid}"/>

<!-- Up to 5 reconnection attempts -->
<action application="audio_fork"
        data="ws://127.0.0.1:8000/ws/${uuid} max_reconnects=5"/>

<!-- Infinite reconnection (production resilience) -->
<action application="audio_fork"
        data="ws://127.0.0.1:8000/ws/${uuid} max_reconnects=-1"/>
Blocking behavior The audio_fork app blocks the dialplan thread for the duration of the call (it drives the read_frame loop). Any dialplan actions after audio_fork will only execute once the call ends.

AudioPipeConfig

The C struct AudioPipeConfig is populated inside mod_audio_fork.c before calling ap_create(). Integrators building on top of the module can extend this structure to pass additional configuration to the pipeline. 

typedef struct {
    const char* metadata_json;          /* [N1] First WS TEXT frame. NULL = none. */
    const char* extra_headers;          /* [N2] HTTP headers on WS handshake.     */
    int         max_reconnect_attempts; /* [M2] 0=none, N=max, -1=infinite.       */
} AudioPipeConfig;
FieldDescription
metadata_jsonJSON string sent as a TEXT frame immediately after the WebSocket is established, before any audio. Pointer must remain valid until AP_EVENT_CONNECTED.
extra_headersAdditional HTTP headers appended to the WS Upgrade request. One header per line, terminated with \n (LWS adds CRLF internally). Pointer must remain valid until AP_EVENT_CONNECTED.
max_reconnect_attemptsControls automatic reconnection. See Reconnection.

Reconnection

When the WebSocket connection is lost and max_reconnect_attempts != 0, audio_pipe will attempt to reconnect using exponential backoff:

  • First retry: 100 ms
  • Each subsequent retry doubles the delay, capped at 5 000 ms.
  • On each attempt, AP_EVENT_RECONNECTING is fired — the call remains active.
  • If all attempts are exhausted, AP_EVENT_DISCONNECTED fires and the call is hung up.
  • With max_reconnects=-1, reconnection continues indefinitely until the call is otherwise terminated.
Audio gap during reconnection The TX path plays silence while the bot is disconnected. The RX path continues to capture caller audio but it is not transmitted. Audio captured during the reconnection window is dropped.

Custom WebSocket Headers

Extra HTTP headers can be injected into the WebSocket Upgrade request via AudioPipeConfig.extra_headers. This is implemented using the libwebsockets LWS_CALLBACK_CLIENT_APPEND_HANDSHAKE_HEADER callback with lws_add_http_header_by_name(). 

/* Example: Bearer token + tenant ID */
const char *headers =
    "Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...\n"
    "X-Tenant-ID: acme\n";

AudioPipeConfig cfg = {
    .metadata_json          = metadata_buf,
    .extra_headers          = headers,
    .max_reconnect_attempts = 5,
};
Separator format Use \n (LF only) between headers — do not use CRLF. libwebsockets adds the proper HTTP separators internally.

API Commands

In addition to the dialplan application, the module exposes two ESL/API commands.

uuid_audio_fork_stop

Stops the active fork by hanging up the channel with NORMAL_CLEARING; teardown then closes WebSocket and removes the media bug.

uuid_audio_fork_stop <uuid>

uuid_audio_fork_stats

Returns a JSON object containing production metrics for the active fork session. See Metrics for field descriptions.

uuid_audio_fork_stats <uuid>
{
  "tx_underruns":   0,
  "tx_ring_drops":  0,
  "sq_drops":       0,
  "ws_reconnects":  1,
  "rx_frag_bytes":  40320
}

Environment toggle

AF_RX_CNG_BRIDGE_ENABLE=0

When set to 1, enables optional RX CNG bridging logic in ap_on_rx_frame(). Default 0 is recommended for natural speech.

Metrics

The AudioPipeStats struct is populated in real time by the pipeline. Stats can be retrieved at any time via uuid_audio_fork_stats or read programmatically via ap_get_stats().

FieldTypeDescription
tx_underrunsuint64Number of 20 ms frames where the TX ring buffer was empty and silence was played instead of bot audio.
tx_ring_dropsuint64Number of bot audio frames dropped because the TX ring buffer was full (back-pressure).
sq_dropsuint64Number of RX frames dropped because the send queue was full (bot ingestion too slow).
ws_reconnectsuint64Total number of WebSocket reconnection attempts made during this session.
rx_frag_bytesuint64Total bytes accumulated in the RX fragment reassembly buffer (split WS frames from the bot).

Stats can be reset mid-call with ap_reset_stats(ap). This is useful for per-utterance latency analysis.

Build & Docker

The module is compiled as part of a Docker multi-stage build based on drachtio/drachtio-freeswitch-mrf. The build stage installs the apt-provided libwebsockets-dev — not the static libwebsockets.a shipped by drachtio — because the drachtio version was compiled without -fPIC and cannot be linked into a shared object. 

# Stage 1 — builder
FROM drachtio/drachtio-freeswitch-mrf:latest AS builder

RUN apt-get update && apt-get install -y --no-install-recommends \
      build-essential g++ libwebsockets-dev libspeexdsp-dev \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /build
COPY mod_audio_fork_build/mod_audio_fork.c .
COPY mod_audio_fork_build/audio_pipe.cpp   .
COPY mod_audio_fork_build/audio_pipe.h     .

RUN g++ -std=c++14 -fPIC -O2 \
        -I/usr/local/freeswitch/include/freeswitch \
        -I/usr/include \
        -c audio_pipe.cpp -o audio_pipe.o \
 && g++ -std=c++14 -fPIC -O2 \
        -I/usr/local/freeswitch/include/freeswitch \
        -I/usr/include \
        -c mod_audio_fork.c -o mod_audio_fork.o \
 && g++ -shared -fPIC \
        -Wl,-soname,mod_audio_fork.so \
        -o mod_audio_fork.so \
        mod_audio_fork.o audio_pipe.o \
        -lwebsockets -lspeexdsp -lstdc++ -lpthread
Why not -Bstatic -lwebsockets? The libwebsockets.a in /usr/local/lib (from drachtio) was compiled without -fPIC. Linking it into a .so causes relocation R_X86_64_PC32 … can not be used when making a shared object. The apt package (libwebsockets-dev) is compiled with -fPIC and does not require LWS_WITH_EVLIB_PLUGINS, resolving both issues.
Compile mod_audio_fork.c with g++, not gcc Although the file has a .c extension, it references AudioPipe members through the public API and must be compiled as C++. Using gcc results in "invalid use of incomplete typedef 'AudioPipe'" because the C compiler treats the forward declaration as opaque.
